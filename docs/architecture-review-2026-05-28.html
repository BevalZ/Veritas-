Paper Audit / Veritas 项目分析

A. 审查运行流程 Module

Strong

把 `main()` 中阶段 1-5 的业务流程沉淀为 Audit Run Module。CLI、测试、未来 UI 都通过 RunRequest 调用同一 Interface。

Before

flowchart LR Args[argparse args] --> Main[main 570 行阶段流程] Main --> Many[全局配置 / HTTP / 缓存 / 报告]

After

flowchart LR CLI[CLI Adapter] --> Run[Audit Run Module] Tests[Integration Tests] --> Run Run --> Result[RunResult + artifacts]

B. 外部依赖 Adapter 与 Critical Capability Policy

Strong

LLM、MinerU、Crossref/OpenAlex/PubMed、图像语义模型、imagedetector 目前直接依赖全局配置和 requests。应抽出 Adapter，并把关键能力 preflight、内容条件阻断、失败诊断报告变成显式运行策略。

Before

flowchart TD Functions[业务函数] --> Globals[全局 API key / URL] Functions --> Requests[requests / urllib] Functions --> Print[print 日志]

After

flowchart TD Run[Audit Run Module] --> Policy[Critical Capability Policy] Run --> LLMAdapter[LLM Adapter] Run --> MinerUAdapter[MinerU Adapter] Run --> RefAdapter[Reference Adapter] Run --> VisionAdapter[Vision Adapter]

C. 证据与报告数据模型

Strong

当前报告、检查项、参考文献和图像结果都用 dict 传递。建议建立 Evidence Model Module，让解析、合并、渲染之间有稳定模型。

Before

flowchart LR LLMJson[LLM JSON/dict] --> Merge[合并 dict] Merge --> Render[HTML/Markdown 读 dict] Ref[参考文献 dict] --> Render Img[图像 dict] --> Render

After

flowchart LR Raw[LLM/Ref/Image Raw] --> Normalize[Evidence Model Normalizer] Normalize --> Model[AuditReport Model] Model --> Renderers[Markdown / HTML / JSON]

D-F. 输入、缓存、报告渲染深化

Worth exploring
  • Input Project Module: 主论文、补充材料、数据文件和参考文献文件分类应形成 InputManifest。
  • Run Workspace Module: 断点缓存、阶段事件、JSON 原子写和失败状态应集中。
  • Report Renderer Module: HTML 皮肤、证据排序、动作面板和内容模型应分层。

Top Recommendation

先处理外部依赖 Adapter、关键能力 preflight 和失败诊断报告，再抽 Audit Run Module。原因是该工具以公开发表文献为对象，质量主要来自第三方服务的完整覆盖；外部 Adapter 抽出后，端到端运行测试和后续流程 Module 都会更稳。
